# DEV.Rapha Portfolio 2.0 Identity Design

Date: 2026-06-14

## Goal

Reposition the DEV.Rapha portfolio and CV package from a junior/front-end portfolio into a premium Quality Engineering, Automation, and Operational Intelligence identity for QA/Automation/Quality Engineering contractor opportunities.

The final system must communicate engineering maturity, operational systems thinking, enterprise troubleshooting, automation mindset, Playwright competence, reliability focus, and AI-native QA workflows.

This is a combined identity package:

- Portfolio 2.0.
- CV PT-BR HTML and PDF.
- CV EN HTML and PDF.

The portfolio is incomplete unless the CVs match the same professional positioning and visual language.

## Target Audience

Primary audience:

- International recruiters.
- Hiring managers.
- QA and engineering leads looking for PJ/contractor profiles.
- Teams hiring for QA Automation, Quality Engineering, Playwright Automation, and technical support-to-QA transition roles.

Secondary audience:

- Brazilian recruiters and local technical contacts.

The site should not optimize for freelance website clients, landing page work, or generic front-end services.

## Approved Direction

Use the **Operational Intelligence Platform** direction.

The portfolio should feel like a modern technical product surface: an operational hub around Quality Engineering, Automation, Reliability, and QA systems. It should not feel like a personal landing page with decorative cards.

Use selected elements from a case-driven direction for the projects section, where each project is framed as an engineering system or validation workflow.

## Positioning

Core positioning:

- Quality Engineer.
- Playwright Automation.
- Operational Systems.
- QA Architecture.
- Enterprise Troubleshooting.
- Reliability Validation.
- API and workflow validation.
- AI-assisted QA workflows.

Primary English positioning line:

> Quality Engineering, Automation & Operational Intelligence

Primary Portuguese positioning line:

> Engenharia de Qualidade, Automação e Inteligência Operacional

Avoid these old signals:

- Front-end beginner.
- Generic web developer.
- Landing pages and sales websites.
- Passionate about coding.
- Junior developer language.
- Generic freelancer services.

## Information Architecture

### 1. Hero / Command Center

Purpose:

Make the first viewport clearly signal QA/Automation/Quality Engineering, not front-end development.

Content:

- Strong headline around automation, reliability, and operational systems.
- Short statement about building reliable workflows, validating complex systems, and improving quality operations.
- CTA group:
  - Primary: Download CV, resolved by active language.
  - Secondary: LinkedIn.
  - Tertiary: Contact.
- Subtle technical signals: Playwright, QA Engineering, Automation, Operational Systems, API Validation, AI workflows.

Visual:

- Left: headline, supporting copy, CTAs, credibility tags.
- Right: abstract operational panel with workflow lines, validation states, automation stack labels, and reliability indicators.
- Profile photo should not dominate the hero. If used, place it in a smaller identity/about context.

### 2. Operational Profile / About

Purpose:

Explain Raphael's profile as a systems-focused QA professional.

Content themes:

- Troubleshooting.
- Enterprise systems.
- Functional validation.
- QA workflows.
- Automation mindset.
- Collaboration with development and product teams.
- Reliability and user-impact thinking.

Tone:

- Professional.
- Technical.
- Operational.
- Direct.

Avoid generic self-description and motivational language.

### 3. Automation Engineering

Purpose:

This is a central section, not a secondary skill block.

It must show the automation engineering direction through concrete domains:

- Playwright E2E.
- Regression automation.
- API validation.
- Cross-browser validation.
- Session/auth validation.
- Release readiness.
- Execution analysis.
- Traceability.
- AI-assisted QA workflows.

Visual:

- Premium engineering section with a structured grid or pipeline, not generic service cards.
- Use operational language such as validation layers, workflow coverage, reliability signals, and release confidence.

### 4. Technical Ecosystem

Purpose:

Reorganize stack around QA and systems work.

Suggested groups:

- QA & Analysis: Functional Testing, Exploratory Testing, Bug Investigation, Regression Testing, Test Planning.
- Automation: Playwright, JavaScript, TypeScript, Node.js, E2E workflows.
- API & Systems: REST APIs, API Testing, Postman, Chrome DevTools, logs, SQL.
- Workflow & Collaboration: Jira, JQL, Git, GitHub, Agile, documentation.
- Web Foundations: HTML, CSS, responsive testing, front-end debugging.

Front-end skills should appear as foundation/supporting skills, not the main identity.

### 5. Systems / Projects

Purpose:

Transform projects into engineering systems and validation cases.

Rules:

- Do not invent technical evidence that the repositories do not support.
- Frame each project by problem, QA/engineering role, stack, and reliability or workflow value.
- Keep GitHub links as proof points.
- Sentinel should be the flagship ecosystem.

Example repositioning:

- Sentinel Tech QA -> QA operational platform / automation initiative / AI-assisted quality system.
- Gestor Web -> Operational Intelligence Dashboard.
- Checklist App -> Operational Workflow Validation System.
- Playwright + k6 -> E2E and Performance Validation Suite. It must not ship with a broken image. If no screenshot is available, use a native system-style project panel instead of an image thumbnail.
- Cromo e Ruinas -> reduce emphasis and frame as a secondary web foundation project, not a core QA proof point.

### 6. Experience

Purpose:

Rewrite professional experience around systems, troubleshooting, reliability, operations, and enterprise workflow validation.

LINX must feel enterprise-level and technically mature:

- Incident analysis.
- L4 support.
- Corporate applications.
- System behavior validation.
- Bug reproduction.
- Log analysis.
- Collaboration with development/architecture teams.
- Critical flow validation before releases or fixes.

Brooks and Kairus should connect operational discipline to QA systems thinking without overstating software engineering claims.

### 7. CV + Contact

Purpose:

Create a coherent conversion path for contractor opportunities.

CTA behavior:

- Active PT-BR site opens PT-BR PDF CV.
- Active EN site opens EN PDF CV.
- Web CV version is available as a secondary link.
- LinkedIn opens in a new tab.
- Contact scrolls to the form.

Contact language:

- Focus on QA/Automation/Quality Engineering opportunities.
- Avoid "Contratar" as a generic service/freelance label. Prefer "Contato", "Falar sobre oportunidade", or English equivalents such as "Discuss an opportunity".

## Visual System

Portfolio style:

- Premium enterprise.
- Dark neutral base.
- Graphite, soft black, slate.
- Deep blue accents.
- Controlled purple accents.
- Green only for validation/reliability signals.
- High readability.
- Minimal, modern operational UI.

Avoid:

- Excessive gradients.
- Cyberpunk/gaming style.
- Heavy glow.
- Visual noise.
- Oversized rounded cards everywhere.
- Generic animated portfolio effects.

Typography:

- Use the shared font stack `Inter`, `Aptos`, `Segoe UI`, system-ui, sans-serif for portfolio and CV files.
- Avoid keeping the visual tone of Poppins if it makes the site feel like a generic developer portfolio.
- Use strong hierarchy, compact UI labels, and readable body copy.

Component language:

- Buttons should be compact and purposeful.
- Tags/chips should show stack and domains.
- Project cards should have technical case anatomy.
- Sections should be broad bands or open layouts with constrained inner content.
- Avoid cards inside cards.

CV visual system:

- CVs remain light and print-first.
- They should reuse the same typography, accent logic, chips, section titles, and positioning language.
- The CV does not need to be dark, but it should feel like the print counterpart of the portfolio.

## Content and Localization

The site remains bilingual:

- PT-BR.
- EN.

The translation architecture should be ready for future additional languages, but only PT-BR and EN are in scope now.

Rules:

- EN copy must be native-sounding, not literal PT-BR translation.
- PT-BR copy should be professional and technical.
- Keep established market terms in English when appropriate, such as Playwright, API Testing, E2E, QA, release readiness.
- Update `html.lang` on language changes.
- The active language controls CV links.
- Update labels, placeholders, ARIA labels, project text, and contact copy.

CV corrections:

- CV EN must use `lang="en-US"`.
- CV EN must not contain Portuguese bullet text except proper names.
- CV PT-BR must not contain accidental English section leftovers unless they are accepted market terms.
- Remove duplicated chips such as repeated JavaScript.
- Fix title/subtitle consistency.
- Normalize location spelling per language.

## Technical Design

The portfolio remains a static GitHub Pages-compatible site.

Expected structure:

- `index.html`.
- `style.css`.
- `script.js`.
- Existing image assets under `src/img/`.
- Published CV assets under `src/cv/`.

The final paths should support:

- PT-BR PDF CV.
- EN PDF CV.
- PT-BR web CV.
- EN web CV.

The JavaScript content model should cover the full page:

- Nav.
- Hero.
- Operational profile.
- Automation engineering.
- Technical ecosystem.
- Projects.
- Experience.
- Contact.
- Footer.
- CV links.

The language switch should update both visible copy and behavior.

## Interactions

Allowed:

- Subtle hover states.
- Smooth transitions.
- Premium motion with restrained glow.
- Mobile nav with a clean transition.
- Active section/nav state.
- Project category controls only if they support the new systems grouping. They must not preserve old labels such as "Web Page" as primary categories.

Avoid:

- Excessive particles.
- Gaming HUD.
- Noisy animation.
- Large motion that distracts from credibility.

## Accessibility and Performance

Requirements:

- Keep visible focus states.
- Preserve form labels or accessible labels.
- Maintain contrast in dark theme.
- Avoid text overlap on mobile.
- Avoid layout shift from images.
- Ensure responsive hero, grids, projects, CV CTA, and contact form.
- Avoid console errors.
- Keep the site static and fast.

## Validation Plan

Before implementation is considered complete:

- Verify desktop layout.
- Verify mobile layout.
- Verify PT-BR and EN language switching.
- Verify CV PDF link follows active language.
- Verify web CV link follows active language.
- Verify LinkedIn opens externally.
- Verify GitHub/project links open externally.
- Verify contact form fields and labels.
- Verify mobile menu.
- Verify no broken images, especially the Playwright screenshot/card.
- Verify browser console has no errors.
- Render/review PDFs from corrected CV HTML files.

## Risks and Mitigations

Risk: The site becomes abstract or buzzword-heavy.

Mitigation: Tie each section to concrete workflows, tools, systems, and experience.

Risk: The dark enterprise look becomes too heavy.

Mitigation: Use generous spacing, thin dividers, clean type, and restrained accent color.

Risk: Portfolio and CV diverge.

Mitigation: Use shared positioning, typography, section logic, chips, and vocabulary.

Risk: The CV EN keeps Portuguese text.

Mitigation: Explicitly audit and rewrite all English CV bullets.

Risk: Project claims overstate current repositories.

Mitigation: Frame projects as validation systems only where evidence exists. Use careful language for foundation projects.

## Out of Scope

- Adding more languages beyond PT-BR and EN.
- Backend services.
- ATS integration.
- Automated job application workflows.
- New project repositories.
- Claims about automation coverage not supported by existing work.

## Approved Decisions

- Direction: Operational Intelligence Platform.
- Target: QA/Automation/Quality Engineering PJ/contractor roles.
- Site remains bilingual PT-BR/EN.
- CV CTA follows active site language.
- PDF CV is primary; web CV is secondary.
- Portfolio and CVs are delivered as a unified identity package.
- The CVs may be updated as needed to align with the new identity.
